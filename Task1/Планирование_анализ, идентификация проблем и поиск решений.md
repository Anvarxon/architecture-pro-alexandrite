# Планирование: анализ, идентификация проблем и поиск решений

## Контекст и допущения

Анализ выполнен по описанию системы и C4-диаграмме контейнеров. Ключевой бизнес-процесс - прохождение заказа через онлайн-магазин, MES, CRM, RabbitMQ, S3-хранилище 3D-моделей и внешних API-пользователей. Там, где есть выбор нового технологического стека, предлагается использовать .NET 8, ASP.NET Core и C#.

## Существующие и потенциальные проблемные места

1. Нет единого технического владельца жизненного цикла заказа. Статусы меняются в разных системах: онлайн-магазин, MES и CRM. При сбое сообщения, ретрая или ручного действия заказ может остаться в промежуточном состоянии без понятного ответственного сервиса.
2. Интеграция MES и CRM через RabbitMQ выглядит недостаточно надежной: из описания не видно transactional outbox/inbox, идемпотентности, DLQ-процесса, повторной обработки и регулярной сверки заказов.
3. Открытие API для партнеров увеличило поток заказов, но не добавило API gateway, квоты, rate limiting, backpressure и договоренности по идемпотентности. Партнеры могут создавать всплески нагрузки и дубли заказов.
4. Расчет стоимости в MES занимает от 2-3 до 30 минут. Если он выполняется синхронно или в общей очереди без приоритетов и контроля SLA, это блокирует поток заказов и ухудшает клиентский опыт.
5. MES отвечает и за расчет стоимости, и за операторский дашборд, и за смену производственных статусов. Эти нагрузки конкурируют за один инстанс приложения и одну БД.
6. Все приложения развернуты по одному инстансу. Нет горизонтального масштабирования, auto healing и независимого масштабирования воркеров расчета.
7. Базы данных имеют по одному инстансу на чтение и запись. Нет read replica для тяжелых отчетных/дашбордных запросов и нет явного разделения OLTP и read model.
8. Первая страница MES медленная даже после фильтра и пагинации. Вероятные причины: плохие индексы, сортировка по большим таблицам, JOIN по статусам, отсутствие материализованного представления/проекции для активных заказов, конкуренция с расчетом стоимости.
9. Операторам нужны самые новые заказы, потому что это влияет на оплату. Текущая архитектура создает бизнес-риск: медленный список заказов напрямую влияет на справедливость распределения работы.
10. Наблюдаемость ограничена Яндекс Метрикой, которая покрывает в основном web/B2C-вход. Нет полной картины по API-партнерам, очередям, базам, расчетам стоимости и статусным переходам.
11. Нет сквозного трейсинга заказа. Команда не может быстро ответить, где завис заказ: в загрузке файла, расчете, очереди, CRM, MES или ручной операции.
12. Нет централизованного структурированного логирования. Разбор инцидентов идет со слов клиента, что увеличивает MTTR и нагрузку поддержки.
13. Нет явных SLO/SLA по этапам заказа: время до расчета цены, время до подтверждения производства, время нахождения в каждом статусе, время до отгрузки.
14. CI/CD зависит от ручного E2E-тестирования. Рост функциональности увеличивает задержку релизов, а исправления инцидентов медленно доходят до production.
15. Окружение release не описано как production-like по данным и нагрузке. Команда может не видеть деградацию производительности до production.
16. В команде только один C#-разработчик, при этом MES - C#-система и главная точка перегрузки. Это риск bus factor и скорости исправлений.
17. Нет отдельного механизма сверки и восстановления заказов: периодическая reconciliation-задача могла бы находить заказы без ожидаемого следующего события.
18. Нет отдельного API/портала для партнеров со статусами, webhooks и прозрачными ошибками. Поэтому партнеры ежедневно обращаются вручную.
19. Нет описанного механизма защиты от дублей сообщений и повторных запросов: idempotency key, уникальные бизнес-ключи, deduplication table.
20. Нет производственного runbook: что делать при росте DLQ, зависании очереди, деградации БД, падении расчета стоимости или массовых жалобах.

## Инициативы для устранения нежелательных ситуаций

1. Внедрить observability: метрики, дашборды, алерты, централизованные логи и distributed tracing. До начала остальных изменений зафиксировать baseline по latency, ошибкам, очередям, времени прохождения статусов и количеству зависших заказов.
2. Ввести сквозную модель жизненного цикла заказа: единый `order_id`, `correlation_id`, список допустимых переходов статусов, журнал статусных событий, SLA на каждый этап.
3. Усилить надежность интеграций через RabbitMQ: transactional outbox/inbox, идемпотентные consumer-ы, DLQ, retry policy, poison-message handling, ручной и автоматический reprocess.
4. Добавить регулярную reconciliation-задачу, которая сверяет заказы в Shop DB, MES DB, CRM и RabbitMQ, находит зависшие состояния и создает инциденты/задачи на исправление.
5. Разделить расчет стоимости и пользовательские/операторские API MES: вынести расчет в асинхронный job pipeline с отдельными воркерами и очередью.
6. Добавить API gateway для внешних партнеров: rate limiting, quota per partner, burst control, idempotency key, версионирование контрактов, понятные статусы ошибок.
7. Реализовать партнерский status API и webhooks, чтобы партнеры не обращались в поддержку за состоянием заказа.
8. Оптимизировать MES-дашборд: индексы под фильтр по статусу и сортировку по времени создания, read model для активных заказов, server-side pagination по cursor/keyset, кеширование горячих выборок.
9. Ввести Redis для кеширования read model MES и короткоживущих справочников, но не использовать кеш как источник истины.
10. Подготовить контейнеризацию сервисов и поэтапный переход к Kubernetes для горизонтального масштабирования API и воркеров.
11. Добавить read replica для тяжелых read-only запросов или вынести аналитические/операторские проекции в отдельную БД.
12. Ввести нагрузочное тестирование критичных сценариев: внешний API, расчет стоимости, создание заказа, MES-дашборд, смена статусов.
13. Автоматизировать E2E smoke/regression в CI для критичного пути заказа и сократить зависимость от ручной проверки.
14. Добавить feature flags/canary release для безопасного выката изменений в order pipeline.
15. Создать support/admin-инструмент: поиск заказа по `order_id`, цепочка событий, последнее успешное событие, кнопка безопасного reprocess для разрешенных ролей.
16. Формализовать SLO: например, 95% расчетов цены до 5 минут, 99% API-ответов до 1 секунды для быстрых операций, 0 потерянных заказов, 99.9% успешной доставки сообщений.
17. Усилить команду: нанять или выделить еще одного C#/.NET-разработчика для MES и одного QA automation/SDET для автоматизации регрессии.

## Приоритизация инициатив

Приоритет расставлен по трем критериям: влияние на потерю заказов и контрактов, скорость снижения операционной боли, риск внедрения. Мониторинг выполняется первым: он фиксирует исходное состояние системы и позволяет оценить эффективность каждой следующей инициативы по измеримым показателям, а не только по количеству жалоб.

| Приоритет | Инициатива | Почему сейчас |
| --- | --- | --- |
| P0 | Observability: метрики, логи, трейсинг, алерты | Без измерений команда не видит источник проблем и не может доказать бизнесу эффект исправлений. |
| P0 | Надежный order pipeline: outbox/inbox, идемпотентность, DLQ, retry, reprocess | Основная жалоба - потерянные и зависшие заказы. Это напрямую влияет на выручку и контракты. |
| P0 | Reconciliation зависших заказов | Даже до полной переработки интеграций нужен механизм находить и возвращать заказы в процесс. |
| P1 | Оптимизация MES-дашборда через read model, индексы и кеш | Снижает боль операторов и ускоряет взятие новых заказов в работу. |
| P1 | Асинхронный расчет стоимости с отдельными воркерами | Убирает 2-30 минутные операции из API-пути и позволяет масштабировать тяжелую нагрузку отдельно. |
| P1 | API gateway и квоты для партнеров | Останавливает неконтролируемый рост нагрузки и защищает B2C-поток. |
| P2 | Партнерские status API/webhooks | Снижает ручные обращения и делает ожидания партнеров прозрачными. |
| P2 | Автоматизация E2E и performance-тестов | Ускоряет релизы и уменьшает регрессии в критичном пути. |
| P2 | Контейнеризация и Kubernetes | Дает масштабирование и auto healing, но требует подготовки observability и деплой-процесса. |
| P3 | Read replica/отдельное хранилище аналитики | Полезно, если оптимизация запросов и read model не снимут нагрузку. |

## Целевая архитектура через полгода

Через полгода целевая архитектура должна быть не полной микросервисной перестройкой, а стабилизированной эволюцией текущей системы:

1. Заказ проходит по управляемому event-driven pipeline. Все события имеют `order_id`, `correlation_id`, `trace_id`, версию схемы и идемпотентный ключ.
2. Интеграции через RabbitMQ используют outbox/inbox, DLQ, retry и reprocess. Потеря сообщения не приводит к потере заказа.
3. Расчет стоимости выполняется асинхронно: API принимает заказ, возвращает идентификатор, а расчет обрабатывается отдельными MES workers. Для новых компонентов используется .NET 8.
4. Внешние партнеры работают через API gateway с квотами, лимитами, статусным API и webhooks.
5. MES-дашборд читает данные из оптимизированной read model активных заказов. Горячие выборки кешируются в Redis, БД остается источником истины.
6. Есть единый observability-слой: Prometheus/Yandex Monitoring, Grafana, Alertmanager, OpenTelemetry Collector, Jaeger/Tempo, OpenSearch/ELK для логов.
7. В production настроены SLO-дашборды: здоровье API, очередей, БД, расчетов стоимости, статусных переходов и просроченных заказов.
8. Релизный процесс включает автоматические smoke/E2E-тесты критичного пути и нагрузочные проверки для MES/API.
9. У поддержки и администраторов есть инструмент расследования заказа: цепочка событий, трасса, логи и безопасный reprocess.
10. Инфраструктура готова к горизонтальному масштабированию: контейнеры, Kubernetes для новых компонентов и постепенная миграция существующих приложений.

## Три инициативы на ближайшие полгода, если можно выбрать только три

1. Observability: мониторинг, трейсинг, централизованные структурированные логи и алерты.

   Это первый шаг, потому что до изменений нужно зафиксировать baseline: количество зависших заказов, глубину очередей, DLQ, latency API и MES-дашборда, время расчета стоимости и прохождения статусов. Без observability команда будет продолжать искать проблемы вручную и не сможет объективно оценить эффективность остальных инициатив.

2. Надежный order pipeline: outbox/inbox, идемпотентность, DLQ, retry, reprocess и reconciliation.

   После появления измерений нужно устранить главную причину потери денег: заказы, которые зависают, теряются или становятся непрозрачными. Даже если производство справляется, компания не выполнит обещание клиенту без надежной доставки событий и восстановления после сбоев. Эффект оценивается по доле зависших заказов, DLQ, количеству повторных обработок и времени прохождения статусов.

3. Производительность и управляемость MES/API: асинхронный расчет стоимости, оптимизированный MES-дашборд, read model, кеш и API gateway с квотами.

   Эта инициатива закрывает две самые заметные боли: партнеры ждут заказы и операторы не могут быстро работать с новыми заказами. Асинхронный расчет и отдельные воркеры снижают влияние тяжелых 3D-моделей на API, а read model и кеш ускоряют первую страницу MES.

Эти три пункта выполняются последовательно и закрывают цепочку "измерить исходное состояние -> не потерять заказ -> выдержать рост нагрузки -> измерить результат". Kubernetes, read replicas и расширение команды важны, но без observability и надежного pipeline они не гарантируют исправления клиентской боли.
