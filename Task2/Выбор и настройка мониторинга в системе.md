# Выбор и настройка мониторинга в системе

## Мотивация

Яндекс Метрика показывает поведение пользователей онлайн-магазина, но после открытия партнерского API этого недостаточно. Часть заказов приходит без web-сессии, а ключевые проблемы находятся внутри backend-сервисов, RabbitMQ, БД, S3 и MES-расчета стоимости.

Мониторинг нужен компании по следующим причинам:

1. Быстро видеть деградацию до массовых жалоб: рост очередей, DLQ, HTTP 500, latency, ошибок расчета цены.
2. Управлять SLA заказа: понимать, сколько заказов зависло в каждом статусе и сколько времени заказ проводит между статусами.
3. Защитить B2C и production от всплесков партнерского API через лимиты и алерты.
4. Дать бизнесу прозрачную картину: где bottleneck - производство, расчет цены, CRM, MES, очередь или БД.
5. Сократить MTTR и поддержку: вместо ручного опроса клиента команда видит технические симптомы и историю деградации.

Ожидаемый бизнес-эффект: меньше просроченных заказов, меньше потерянных контрактов, меньше ручных обращений в поддержку, быстрее релизы исправлений и понятное планирование масштабирования.

## Выбор подхода к мониторингу

Используется комбинированный подход:

1. RED для API и сервисных методов: Rate, Errors, Duration. Подходит для Shop API, CRM API, MES API и партнерского API-входа.
2. USE для инфраструктуры: Utilization, Saturation, Errors. Подходит для EC2/VM, БД, RabbitMQ, S3, CPU, memory, disk, connections.
3. Четыре золотых сигнала для пользовательского и бизнес-пути: latency, traffic, errors, saturation. Подходит для end-to-end процесса заказа и SLO.

Такой выбор лучше, чем один универсальный подход, потому что система смешанная: есть HTTP API, очереди, долгие фоновые расчеты, БД и бизнес-статусы заказа.

## Метрики

### API-метрики

| Метрика | Где | Зачем | Ярлыки |
| --- | --- | --- | --- |
| Number of requests (RPS) for internet shop API | Shop API | Видеть трафик B2C и влияние маркетинга/сезонности. | `service`, `env`, `method`, `route`, `status_code`, `client_type=b2c` |
| Number of requests (RPS) for CRM API | CRM API | Видеть нагрузку продавцов и внутренних процессов. | `service`, `env`, `method`, `route`, `status_code`, `role` |
| Number of requests (RPS) for MES API | MES API | Видеть нагрузку операторов и партнеров на расчет/статусы. | `service`, `env`, `method`, `route`, `status_code`, `client_type`, `partner_id` |
| Number of requests (RPS) per user for CRM/MES/API | CRM API, MES API, API Gateway | Находить аномальных пользователей/партнеров и вводить квоты. | `service`, `env`, `user_id_hash`, `partner_id`, `route` |
| Response time (latency) for shop/CRM/MES API | Все backend API | Контроль p50/p95/p99 и влияния на UX. | `service`, `env`, `method`, `route`, `status_code` |
| Number of HTTP 200 for shop/CRM/MES API | Все backend API | Базовая оценка успешных ответов и трафика. | `service`, `env`, `route`, `method` |
| Number of HTTP 500 for shop/CRM/MES API | Все backend API | Раннее обнаружение инцидентов. | `service`, `env`, `route`, `method`, `exception_type` |
| HTTP 4xx rate | API Gateway, все backend API | Видеть ошибки контрактов, авторизации, rate limit и валидации. | `service`, `env`, `route`, `status_code`, `partner_id` |
| Number of simultaneous sessions | Shop, CRM, MES | Понимать нагрузку UI и активность операторов/продавцов. | `application`, `env`, `role` |
| Kb transferred/provided | Все backend API | Контроль сетевой нагрузки, больших загрузок и ответов. | `service`, `env`, `route`, `direction`, `client_type` |

### RabbitMQ-метрики

| Метрика | Зачем | Ярлыки |
| --- | --- | --- |
| Number of dead-letter-exchange letters in RabbitMQ | Любое значение выше нуля означает, что часть событий не обработана штатно. Для заказов это P0/P1 риск. | `env`, `vhost`, `exchange`, `queue`, `routing_key`, `consumer_service` |
| Number of message in flight in RabbitMQ | Рост unacked/in-flight показывает, что consumer-ы не успевают или зависают. | `env`, `queue`, `consumer_service` |
| Queue depth | Видеть backlog по созданию заказов, расчету цены и статусным событиям. | `env`, `queue`, `message_type`, `priority` |
| Publish/consume rate | Сравнить входящий и исходящий поток, увидеть деградацию consumer-ов. | `env`, `queue`, `exchange`, `consumer_service` |
| Retry count and poison messages | Отличать временные ошибки от сообщений, которые всегда падают. | `env`, `queue`, `message_type`, `exception_type` |

### Ресурсные метрики

| Метрика | Где | Зачем | Ярлыки |
| --- | --- | --- | --- |
| CPU % for shop/CRM/MES API | EC2/VM/container | Понимать насыщение API и необходимость масштабирования. | `service`, `env`, `instance` |
| Memory Utilisation for shop/CRM/MES API | EC2/VM/container | Находить memory leak и пределы масштабирования. | `service`, `env`, `instance` |
| Memory Utilisation for shop db/MES db instance | Managed DB | Контроль деградации БД и кеша страниц. | `db`, `env`, `instance` |
| Number of connections for shop/MES db instance | Managed DB | Защититься от исчерпания пула соединений. | `db`, `env`, `application` |
| DB CPU, IOPS, locks, slow queries | Shop DB, MES DB | Обязательное дополнение к списку: именно БД может тормозить MES-дашборд. | `db`, `env`, `query_fingerprint`, `table` |
| Size of S3 storage | 3D files storage | Планирование затрат и контроль роста 3D-файлов. | `bucket`, `env`, `file_type` |

### Бизнес-метрики заказа

| Метрика | Зачем | Ярлыки |
| --- | --- | --- |
| Orders created per minute | Видеть реальную нагрузку, включая B2C и партнеров. | `env`, `source=b2c|partner`, `partner_id` |
| Orders by status | Видеть накопление заказов на каждом этапе. | `env`, `status`, `source`, `partner_id` |
| Order status transition duration | Главная SLA-метрика: сколько времени занимает переход между статусами. | `env`, `from_status`, `to_status`, `source`, `model_complexity_bucket` |
| Stuck orders count | Количество заказов, которые дольше порога находятся в статусе без движения. | `env`, `status`, `source`, `age_bucket` |
| Price calculation duration | Контроль тяжелого MES-процесса. | `env`, `source`, `model_complexity_bucket`, `polygon_count_bucket` |
| Price calculation queue wait time | Понимать, проблема в вычислении или ожидании очереди. | `env`, `priority`, `source` |
| MES dashboard load time | Контроль боли операторов. | `env`, `status_filter`, `page_size`, `sort` |

Важно избегать высококардинальных labels с персональными данными. `user_id` и `partner_id` хранить как технические идентификаторы или хеши. `order_id` не добавлять как label в Prometheus; его использовать в логах и трассах.

## План действий

1. Определить SLO и error budget для критичных процессов: создание заказа, расчет цены, подтверждение производства, отображение MES-дашборда, доставка события через RabbitMQ.
2. Развернуть Prometheus-compatible time-series хранилище: Managed Service for Prometheus/Yandex Monitoring для облачных ресурсов и Prometheus для приложений, если managed-вариант не покрывает все exporters.
3. Развернуть Grafana и Alertmanager. Создать дашборды для API, RabbitMQ, БД, MES-дашборда и бизнес-статусов заказа.
4. Подключить exporters: RabbitMQ exporter, PostgreSQL exporter, node/container exporter, cloud monitoring exporter для managed services.
5. Инструментировать Java Spring Boot сервисы через Micrometer/OpenTelemetry, MES на C# через OpenTelemetry .NET SDK.
6. Ввести единый набор labels: `service`, `env`, `route`, `method`, `status_code`, `client_type`, `partner_id`, `source`, `status`.
7. Добавить бизнес-метрики в backend: создание заказа, статусные переходы, время расчета, зависшие заказы.
8. Настроить алерты P0/P1/P2 и каналы: мессенджер команды, email продукту для бизнес-деградаций, дежурная эскалация для P0.
9. Добавить runbook к каждому алерту: что проверить, где смотреть дашборд, как reprocess-ить сообщение, когда включать rate limit.
10. Добавить мониторинг в release-окружение и использовать его для performance-gate перед production.

## Показатели насыщенности и реакции

| Показатель | Warning | Critical | Почему такой порог | Действие |
| --- | --- | --- | --- | --- |
| CPU API/worker | >70% 15 минут | >85% 10 минут | После 70% уменьшается запас на всплески, после 85% растет latency. | Warning: тикет на capacity. Critical: алерт дежурному, масштабирование/перезапуск, анализ профиля. |
| Memory API/worker | >80% 15 минут | >90% 10 минут или OOM | Риск GC pressure/OOM и деградации. | Алерт, heap dump при возможности, rollback при росте после релиза. |
| DB connections | >70% пула | >85% пула | Исчерпание пула приводит к массовым таймаутам API. | Проверить утечки соединений, увеличить pool/лимит только после анализа. |
| DB CPU/IOPS | >70% 15 минут | >85% 10 минут | БД - общий bottleneck для заказов и MES. | Проверить slow queries, индексы, временно ограничить тяжелые запросы. |
| RabbitMQ DLQ | >0 сообщений 5 минут | >10 сообщений или любое P0-событие | DLQ для заказа означает нештатную обработку. | Создать incident/ticket, включить reprocess после анализа причины. |
| RabbitMQ queue depth | >5 минут расчетной емкости consumer-ов | >15 минут расчетной емкости | Важна не абсолютная глубина, а сколько времени потребуется на разбор backlog. | Warning: добавить consumer/воркер. Critical: включить rate limit партнеров, эскалация. |
| In-flight/unacked messages | >1000 или растет 10 минут | Растет 30 минут без consume progress | Признак зависших consumer-ов или долгих обработок. | Перезапуск consumer-ов, анализ зависаний, временное снижение prefetch. |
| API p95 latency | >1 сек для быстрых API | >3 сек 10 минут | Быстрые операции не должны ждать расчет 3D-модели. | Проверить route, БД, downstream. Для партнеров включить backpressure. |
| Price calculation p95 | >5 минут | >15 минут | Нормальный средний расчет 2-3 минуты; >15 минут приближается к худшему сценарию. | Масштабировать воркеры, проверить сложность моделей, приоритеты и очередь. |
| MES dashboard p95 | >2 сек | >5 сек | Операторы должны быстро видеть новые заказы. | Проверить read model/cache/индексы, отключить тяжелые фильтры при инциденте. |
| Stuck orders count | >0 заказов старше SLA | >10 или VIP/крупный партнер | Любой зависший заказ влияет на SLA. | Создать задачу поддержки, автоматическая reconciliation, уведомить менеджера. |

При превышении critical-порогов должна создаваться incident-запись с владельцем, ссылкой на дашборд, последними релизами и runbook. Для партнерского API дополнительно включается автоматическое ограничение burst-нагрузки и возврат `429 Too Many Requests` с `Retry-After`.
